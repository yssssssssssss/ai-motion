# 变更提案: 帧间还原能力产品化增强

## 需求背景
Motion Copilot 的「帧间」已经具备 legacy FrameSnapshot 低保真编译、高保真 ZeroVisualSnapshot 渲染、节点绑定、时间轴生成和导出能力。但真机测试暴露出几个阻碍实际使用的问题：

1. 低保真路径仍可能丢失圆角背景、SVG 背景、阴影、边框、字体等关键视觉信息。
2. 高保真路径视觉更接近设计稿，但编辑体验依赖代理图层，用户不容易理解“看到的节点”和“可编辑图层”的关系。
3. 图层命名、图层层级、节点匹配、进入/退出判断仍需要人工排查，系统没有给出质量诊断。
4. 当前无法明确告诉用户“这次还原是否可信、哪里有风险、下一步该如何修正”。

因此下一阶段目标不是承诺任意设计 100% 还原，而是建立一套可诊断、可修正、可回归的帧间生产链路。

## 产品分析

### 目标用户与场景
- **用户群体:** 动效设计师、交互设计师、前端工程师、AI motion 工具评审者。
- **使用场景:** 从手机 App 设计稿的两个状态帧生成动效，例如状态条展开、卡片展开、按钮组切换、列表状态变化、底部弹层进入退出。
- **核心痛点:** 用户需要快速生成可播放动效，但不能接受关键样式丢失、图层错乱、选中跳位、匹配不可解释。

### 价值主张与成功指标
- **价值主张:** 高保真保底，可编辑增强，问题可诊断，结果可修正。
- **成功指标:**
  - 高保真导入后关键节点保留率达到 98% 以上。
  - M1 阶段：还原报告能覆盖节点缺失、关键样式缺失、层级错乱、低置信度匹配四类问题，并能定位到 nodeId / layerId。
  - M2-P0 阶段：对象模型先覆盖 text / container / asset 三类基础对象，不承诺复杂控件识别准确率。
  - M2-P1 阶段：button / status-pill / label-group 等常见 App UI 控件对象识别准确率再以 90% 以上作为目标。
  - 低置信度匹配必须进入待确认，不静默误配。
  - 真机回归样例中首尾帧关键视觉缺陷归零。
  - 用户能在 3 分钟内定位并修正一次帧间导入的问题。

### 人文关怀
工具不应把“看起来像 AI 生成”置于专业可控性之上。对设计师而言，可解释的失败比不透明的自动猜测更重要；因此系统应明确展示风险，而不是隐藏问题。

## 变更内容
1. 建立帧间还原评分报告，输出结构、样式、层级、匹配、视觉风险等诊断结果。
2. 引入对象级图层模型，把按钮、状态胶囊、标签组、容器等从原始节点中聚合出来。
3. 优化高低保真统一流程：高保真视觉保底，低保真/代理图层用于编辑和时间轴控制。
4. 增强匹配可视化与人工修正能力，支持查看匹配原因、置信度、强制匹配、强制进入/退出、忽略节点。
5. 建立真机回归样本库，将真实反馈样例转成固定 fixture 和测试用例。

## 核心场景

### 需求: 导入后可判断还原质量
**模块:** 帧间评分报告
用户从 Zero 读取首尾帧后，系统生成质量报告。

#### 场景: 关键背景缺失
当 SVG 背景或圆角矩形未进入渲染/绑定链路：
- 报告列出缺失 nodeId、名称、类型和影响范围。
- 报告给出建议动作，例如“保留为 asset 背景”或“转成可编辑 rect”。

### 需求: 节点被组织为用户可理解的对象
**模块:** 对象级图层模型
系统不只展示 `文字 159`、`矩形 5125`，而是识别出按钮和状态胶囊。

#### 场景: 状态条展开
对于“指派域产品 / 状态胶囊 / 继续指派 / 生成PRD / 邀请主架构”：
- 图层列表显示对象名称。
- 用户可以展开查看内部文本、背景、圆点。
- 时间轴可以绑定到对象或子节点。

### 需求: 匹配关系可解释可修正
**模块:** 匹配可视化
系统展示每条匹配为什么成立。

#### 场景: 低置信度文本匹配
当两个节点都叫 `文字 159` 但文本不同：
- 不自动匹配。
- 进入待确认列表。
- 用户可手动指定 matched / enter / exit / ignore。

### 需求: 高保真和可编辑不割裂
**模块:** 高低保真统一链路
高保真 iframe 负责结果层，可编辑代理层负责时间轴与参数。

#### 场景: 点击高保真节点
用户点击 iframe 中的节点：
- 左侧图层对象选中。
- 右侧展示节点来源、绑定关系、可编辑参数。
- 底部时间轴选中对应片段。

## 风险评估
- **风险:** 试图一次性把所有 Zero 节点转成完全可编辑原生图层，会导致复杂度失控。
- **缓解:** 坚持”高保真保底，可编辑增强”，优先让结果层正确，再逐步开放对象级编辑。
- **风险:** 评分报告变成装饰性分数，不能指导修复。
- **缓解:** 每个扣分项必须绑定 nodeId、字段、严重级别、建议动作和回归测试。UI 默认展示 issue 分级汇总（error/warning/info 各几条），总分放二级或隐藏——用户关心的是”哪里有问题”而非具体分值。
- **风险:** UI 面板过载。
- **缓解:** 默认只显示总分和高风险问题，详细匹配/样式诊断放入可展开调试区。
- **风险:** 对象识别（ObjectModel）纯规则在真实 App UI 上准确率难以稳定到 90%。
- **缓解:** P0 阶段只做 text / container / asset 三类纯几何识别；button / status-pill / label-group 归为 P1 阶段用 AI 语义辅助，避免一次性上线不成熟的规则识别。
- **风险:** 匹配手动修正缺少持久化，重新导入或重新打开项目会丢失用户覆盖。
- **缓解:** 在 `packages/motion-copilot-core/src/schema/document.ts` 的 `VisualCompositionSource` 中增加 `userBindingOverrides?: Array<{fromNodeId, toNodeId, action}>` 字段，确保手动修正跟随项目保存。
- **风险:** 新增 `RestorationIssue` 与已有 `MorphIssue` 边界模糊，后续维护混淆。
- **缓解:** 明确定义边界：`MorphIssue` = 时间轴编排阶段问题（track 缺失、timing 冲突），`RestorationIssue` = 导入还原质量问题（样式丢失、节点覆盖不足、层级错乱）。两者并存，分别在不同面板展示。
- **风险:** 35+ 子任务缺少里程碑切片，容易陷入”永远 50% 完成”。
- **缓解:** 定义两个交付里程碑：M1（评分报告 + fixture + 质量面板只读）验收”导入后能看到问题在哪”；M2（对象模型 P0 子集 + 匹配修正 + 联动）验收”能修正并重跑”。
