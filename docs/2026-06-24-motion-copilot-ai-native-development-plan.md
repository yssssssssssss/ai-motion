# Motion Copilot 下一阶段开发文档与 Todo

日期：2026-06-24

范围：独立模块 `apps/motion-copilot` 与 `packages/motion-copilot-core`。本计划不要求合入旧编辑器主流程，也不复用旧系统能力。

## 1. 当前能力基线

Motion Copilot 当前已经具备“可控动效搭建器”的核心底座。它不是静态页面 demo，而是一套可以独立运行、独立测试、独立演示的动效编排模块。

### 1.1 独立模块能力

- 前端应用：`apps/motion-copilot`
- 核心包：`packages/motion-copilot-core`
- 独立开发启动：`pnpm --filter @motion-copilot/app dev`
- 独立测试：`pnpm motion-copilot:test`
- 独立类型检查：`pnpm motion-copilot:typecheck`
- 独立构建：`pnpm motion-copilot:build`

### 1.2 图层与画布能力

- 支持新建空白项目和加载示例项目。
- 支持移动端尺寸、Web 尺寸和自定义画布宽高。
- 支持新增文本图层和图片图层。
- 支持上传背景图、上传前景图片、替换选中图片。
- 支持图层选中、拖拽移动、尺寸调整、重命名、排序、复制、隐藏、锁定、删除。
- 支持文本字号、字重、颜色、行高、对齐方式等基础样式。
- 支持图片适配方式、圆角、透明度、背景色等视觉参数。

### 1.3 规范动效库能力

- 已沉淀一批移动 App 场景动效原子按钮。
- 左侧动效库按“推荐、组合、转场、导航、横向、弹窗、底板、遮罩、卡片、反馈、加载、骨架、列表”等分类组织。
- 支持搜索规范动效。
- 支持基于当前图层展示推荐动效。
- 单个动效可以直接应用到当前图层，也可以加入底部编排时间轴。
- 支持组合预设，例如弹窗进场、底部面板、Tab 切换、加载转成功。
- 支持部分规范约束与冲突提示，例如骨架屏和位移类动效冲突。

### 1.4 多图层时间轴能力

- 支持底部多图层时间轴。
- 支持按图层拆分轨道。
- 支持动效片段拖拽排序、横向调整开始时间、跨轨道改变作用图层。
- 支持时间刻度、片段起止时间窗、吸附线和轨道折叠。
- 支持串行和并行关系。
- 卡片展示“开始时间 + 时长”，不再暴露容易混淆的延迟字段。
- 右侧面板支持编辑片段开始时间、时长、串并行、缓动、起止 X/Y/Scale/Opacity/Blur。

### 1.5 预览与导出能力

- 支持实时画布预览。
- 支持播放、暂停、归零、循环播放。
- 支持基础 HTML/CSS 导出、组合 HTML 导出、编排 JSON 导出和研发参数表 Markdown 导出。
- 支持本地自动保存、清除自动保存、项目导入导出。
- 支持本地 deterministic 结构化生成：从“动效描述”生成可编辑的多图层时间轴草稿。

### 1.6 当前验证状态

- core 测试：30 个用例。
- app 测试：24 个用例。
- 已覆盖图层编辑、上传图片、样式编辑、时间轴拖拽、轨道折叠、组合预设、项目状态和规范约束展示。

## 2. 搭建式方案评估

### 2.1 是否是创建动效的最优解

搭建式方案是当前阶段最稳的解，但不是最终形态的唯一最优解。

动效生产不是单点生成问题，而是图层、时序、规范、节奏、交付格式共同组成的系统问题。搭建式方案把这些信息拆成可编辑对象，让用户能明确控制“谁在动、什么时候动、动多久、和谁一起动、是否符合规范”。这对设计师精修、研发交付和后续 AI 接入都很关键。

但纯搭建式会带来门槛。用户需要理解图层、轨道、开始时间、时长、串并行和规范动效。专业用户可以接受，普通用户会觉得重。因此最终产品形态不应该只有搭建式入口。

更合理的产品路径是：

> AI 生成初稿，搭建式系统负责精修、约束、验证和交付。

### 2.2 搭建式方案的优秀点

- 可控：所有动效都能落到图层、时间轴和参数上。
- 可解释：用户可以看到每个片段的作用图层、开始时间、时长和动效类型。
- 可编辑：AI 或模板生成后的结果仍然能被人工调整。
- 可复用：规范动效和组合预设可以沉淀为可复用资产。
- 可验证：冲突提示、规范约束和测试可以围绕结构化数据建立。
- 可交付：结构化时间轴天然适合导出 JSON、CSS、React 片段和研发参数表。

### 2.3 搭建式方案的限制

- 学习成本高：新用户需要理解轨道和时序。
- 操作成本高：复杂动效需要多次选择图层、添加动效、调整时间。
- 复杂项目可读性会下降：时间轴片段增多后，用户不容易快速理解整体节奏。
- AI 感弱：如果没有自然语言生成入口，发布时容易被认为只是高级编辑器。
- 图层语义有限：当前以文本和图片为主，复杂组件、状态组件、按钮状态和列表子项还不够强。
- 导出仍需增强：当前已支持编排 JSON、研发参数表与 HTML/CSS 验证导出，但 React/CSS 片段还不完整。
- 规范约束还偏局部：需要更多约束规则、原因解释和自动修正策略。

## 3. 下一阶段产品方向

下一阶段不建议继续单纯堆更多动效按钮。重点应该从“可搭建”升级到“可生成、可解释、可交付”。

### 3.1 产品目标

- 让用户用自然语言或场景选择生成动效初稿。
- 让生成结果直接落到多图层时间轴。
- 让用户可以继续精修图层、时间、时长、缓动和参数。
- 让导出结果可以进入研发生产链路。
- 保持 Motion Copilot 作为独立模块，不污染旧系统。

### 3.2 核心产品闭环

```text
用户意图
  -> 结构化理解
  -> 生成动效编排草稿
  -> 规范约束校验
  -> 时间轴可视化精修
  -> 预览与循环观察
  -> 导出研发交付物
```

### 3.3 AI Native 定义

这里的 AI Native 不等于只加一个聊天框。它应该体现为：

- AI 能理解画布中的图层和图层类型。
- AI 能选择符合规范的动效原子。
- AI 能生成多图层时间轴，而不是单个动画。
- AI 能解释为什么这样安排时序。
- AI 的输出可编辑、可回滚、可验证、可导出。

### 验收标准

- 一个已编排项目可以导出稳定 JSON。
- JSON 能被测试用例重新读取并还原为同等时间轴。
- 研发参数表不依赖 UI 文案截图，也能说明动效实现。
- 导出结果中不出现“目标对象 / 主目标 / 选中图层”等旧概念。

### React/CSS 片段导出方案

- 输入：以 `motion-copilot.composition.v1` JSON 为唯一数据源，不直接读取 UI 状态。
- 输出：一个 React 组件片段和一段 CSS keyframes，组件只负责渲染图层结构，CSS 负责时间窗、缓动和起止状态。
- 命名：keyframes 使用 `mc-${stepId}`，图层节点使用 `data-layer-id` 和 `data-motion-step-id` 保持可追溯。
- 约束：首版不引入 Framer Motion 等运行时依赖，避免交付给研发后增加不可控包体。
- 验证：导出前先通过 `validateCompositionJsonExport`；未通过时返回可读错误，不生成片段。

优先级判断：先做导出，再做 AI。原因是 AI 生成也需要稳定的结构化协议承载；没有导出协议，AI 输出很容易变成只服务 UI 的临时状态。

## 6. 风险与应对

### 风险 1：AI 入口做成聊天外壳

应对：先定义结构化输入输出协议，再接模型。模型输出必须落到时间轴和图层上。

### 风险 2：时间轴复杂后不可读

应对：增加并行组、轨道过滤、时间轴缩放和节奏摘要。

### 风险 3：导出与预览不一致

应对：导出结果必须有快照测试和回放测试。

### 风险 4：继续堆动效按钮导致产品变重

应对：动效库继续扩展前，先完善推荐、搜索、AI 生成和约束解释。

### 风险 5：模块边界被旧系统稀释

应对：继续保持 `apps/motion-copilot` 与 `packages/motion-copilot-core` 独立，不把临时实验直接塞进旧编辑器。
